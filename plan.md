「受託開発の“頭の中の状態”を
人間じゃなくてブラウザが覚えていてくれる仕組み」

だよね。

「誰が」「どこまで」「何をやって」「どこで詰まってて」「バグがどう起きたか」
これが口頭やチャット伝達だと絶対漏れるから、“構造化して残したい”って話。

なので、まず このツールが解決すべき本質 を言語化するところからやる。
そのうえで、MVPとして何が必要か を、自問自答スタイルで整理する。

1. このツールのゴールを一言で定義する

受託案件ごとに、「今なにが起きているか」を、誰が見ても 30秒で把握できる状態にするブラウザツール。

もっと分解すると：

案件情報（誰の案件で、いくらで、いつまでか）が一目で分かる

進捗（どこまで終わっていて、何がまだなのか）が明確

誰が最後に触ったか、いつ触ったか、何をしたかが履歴で残る

バグ・トラブルが起きたときに、「そのときの状況」が再現できるレベルで残っている

引き継ぎ時に、前任者の頭の中を引きはがして、画面で共有できる

ここまでできたら、「属人開発」→「チーム開発」に変えられる。

2. 自問自答しながら「必要な機能」を洗い出す

君が頼んだ通り、俺が勝手に問いを立てて、その場で答えていく。

Q1. 1つの案件について、最低限、何を保存しておけばいい？

A. 案件の「名刺部分」と「仕事の設計図」がいる。

① 案件の名刺（ヘッダー情報）

案件名：「某社スクレイピングツール開発」

顧客：会社名／担当者名

金額：見積金額／請求予定金額

契約種別：スポット／月額／準委任 etc

納期：一次納期・本番納期

ステータス：見積中／開発中／検収中／運用中／完了／保留

担当者：

メインエンジニア

補佐／レビュワー

PM（いれば）

② 仕事の設計図（チェックリスト）

案件タイプごとに「テンプレチェックリスト」が必要になる。

例：スクレイピングツール開発 なら：

要件定義

対象サイトの構造調査

スキーマ設計（どのデータをどの形式で出すか）

技術選定（言語／ライブラリ）

プロトタイプ作成

サンプルデータ取得

エラー時の仕様決定（CAPTCHA、429、ブロック etc）

運用方法（実行頻度、環境）

ログ設計

納品形式（コードだけ／実行環境込み etc）

各項目に対して：

状態：未着手／作業中／レビュー待ち／完了／保留

担当者

期限

最終更新日時

が持てれば、「今どこまで？」が視覚化できる。

Q2. 「今どこまで終わったか」をどう可視化する？

A. チェックリスト＋タイムライン＋担当者の組み合わせ。

必要な要素：

各タスク（チェック項目）の状態（ステータス）

“最後に触った人” と “最後に更新した日時”

状態変更の履歴（誰が作業中→完了にしたか）

UIイメージ：

左側：タスクのリスト（チェックリスト）

右側：選んだタスクの詳細（メモ・添付・履歴）

上部：案件のステータスバー（0〜100%など）

Q3. バグが起きたときに、「伝聞」じゃなくて「再現できる情報」を残すには？

A. “Issue（チケット）”的なものが必要。

バグ・トラブルの最低限の項目：

発生日

起きた現象（テキスト）

再現手順（簡易でも良い）

期待する動作

実際の動作

ログ／スクショ／サンプルデータ

対応ステータス：未対応／対応中／確認待ち／クローズ

担当者

原因カテゴリ：

仕様抜け

実装バグ

外部サービス／サイト側変更

想定外データ

これを案件に紐づけて残せば、
「前やらかしたバグ」と「どう直したか」が後から見える。

Q4. 引き継ぎする時に、毎回口頭で聞いてることは何か？

A. たぶんこんな感じになってない？

どこまで終わってる？

どこが難所だった？

どの部分が特に壊れやすい？

仕様であいまいだったところは？

顧客が特に気にしてるポイントは？

ソースコードの「入り口」はどこ？

手動でやらないといけない運用フローは？

これを**「引き継ぎメモ」テンプレ**として画面側に強制する。

引き継ぎタブにこんな項目を持たせる：

この案件の“危険ポイント”

顧客がよく言うキーワード／口癖（細かいけど効く）

暗黙ルール（「ここのサイト、夜間は落ちやすい」みたいなやつ）

ここだけは触るな／ここはリファクタリング予定

コードのエントリーポイント

これが埋まってないと、「引き継ぎ完了」にできない仕組みもアリ。

Q5. 「お金」と「工数」の視点も入れるべきか？

A. 絶対入れた方がいい。経営者として。

ツールの中に最低限ほしい数値情報：

案件単価

想定工数（見積工数）

実績工数（ログベース）

担当者別の実績時間

案件ごとの粗利感（ざっくりでも）

これがあると、

「誰の案件が赤字か」

「どのタイプの案件は二度と取らない方がいいか」

が見えるようになる。

3. ざっくりのデータ構造イメージ（技術者向けの言葉で整理）

これだけあれば、
「今どこ」「誰が」「何を」「どれくらい時間かけた」「どんなバグがあった」が全部拾える。

5. このプロダクトの危険ポイントも正直に言う

君に有利な話だけしても意味がないので、冷静に。

頭の中の全部をツールに押し込もうとすると、現場が面倒くさくて使わなくなる

入力項目が多すぎると、「結局メンテされないデータベース」になる

最初から完璧なデータモデルを目指すと、開発が終わらない

だから、最初のMVPでは「3つだけ」に絞るべきだと考える。

案件ヘッダー情報（顧客／金額／納期／担当）

チェックリスト（状態＋担当＋最終更新者）

簡易Issue（バグ／メモ）

これだけでも、「今どこまで？」問題はかなり軽減する。

6. 次の一手（現実的な進め方）

経営目線でいうと、正しい順番はこう：

まずはこのツールを「自社の受託案件管理」に使う前提でMVPを作る

自分たちで運用して、「項目多すぎ／足りない」を実感で調整

それでもし機能が洗練されたら、
それ自体を“SaaSか半内製ツール”として他社に売る選択肢も出てくる

まとめ：このツールが本当にやりたいこと

君の違和感はこういうことだと思う：

「受託開発の難しさはスキルじゃなくて、
“頭の中の状態管理”を共有できてないことにある」

で、その違和感は完全に正しい。

だからこのブラウザツールは、

「開発者の頭の中の状態」を

「案件」「チェックリスト」「Issue」「引き継ぎメモ」に分解して

人ではなくシステムに覚えさせる

ためのものだ、と定義できる。

まとめ

このブラウザツールは、要するに：

「受託案件ごとに、状態・責任・履歴・暗黙知をすべて
画面上に“固定してしまう”ためのシステム」

1. エンジニアがこだわるべきポイント（先に全体像）

このプロダクトで、エンジニアが特に気にするのはここ：

エンティティ設計

Project / Task / Issue / WorkLog / User / Role / Client / HandoverNote あたりの分解

リレーションの粒度

「誰が」「どの案件で」「どのタスクを」「いつ触ったか」をどこまで正確に持つか

ステータスの定義と遷移ルール

Project.status, Task.status, Issue.status の値と、どういう遷移がOK／NGか

権限・可視性

外注は何が見えて何が編集できるのか（Project, 金額, 顧客情報など）

履歴・監査（Audit）

「誰が変更したか」がどこまで遡れるか（最低限どのエンティティの更新をログるか）

削除 vs アーカイブ

案件・タスク・Issueを本当に消すのか、論理削除か

ここを決めずに「まあやりながら…」をやると、100%地獄行きなので、今ここで言語化しておく。

3. 状態（ステータス）とビジネスルール

ここをガチガチにしておかないと、「仕様書と画面とデータが全部バラバラ」が始まる。

3-1. Project.status

候補（enum）：

draft（作成中・見積り前）

estimating（見積中）

in_progress（開発中）

review（検収中）

operation（運用中）

completed（完了）

on_hold（保留）

canceled（中止）

遷移ルール（例）：

draft → estimating / in_progress / canceled

estimating → in_progress / canceled

in_progress → review / on_hold / canceled

review → operation / completed / in_progress

operation → completed / on_hold

on_hold → in_progress / canceled

※UI 側では「不自然な逆戻り」は選択肢から消した方がいい（例：completed→draft は禁止）。

3-2. Task.status

候補：

todo（未着手）

in_progress（作業中）

review（レビュー待ち）

blocked（ブロック中）

done（完了）

ルール：

進捗バーは done タスク数 / 全タスク で計算

blocked はプロジェクトヘッダーに「⚠ Blockedタスク数」として集計し、上から見えるようにする

done → todo への戻しは Admin / PM のみにするかどうかを決める（現場トラブルの元）

3-3. Issue.status

候補：

open（未対応）

in_progress（対応中）

waiting（顧客確認待ち等）

closed（クローズ）

Issue は比較的シンプルで良い。
大事なのは、優先度と原因カテゴリを入れて、後で「どんな事故が多いか」を見れるようにしておくこと。

3-4. 権限まわり（外注対応）

最小限でこう決めると現実的：

Admin

全案件・全ユーザー・全金額の閲覧＆編集が可能

Member（正社員）

自分が ProjectMember に含まれる案件の閲覧＆編集

金額は閲覧可（会社ポリシー次第）

External（外注）

自分が ProjectMember になっている案件のみ閲覧可

顧客情報の詳細（連絡先・金額）は隠す or マスク

Task / Issue のうち「自分が担当 or 自分所属案件」のみ編集可

Project.status は変更できない

DB 的には role＋ProjectMember.role_in_project の組み合わせで判定して、
フロントでは「押せないボタンは表示しない」が鉄板。

4. デザイン定義（UIコンポーネント & ルール）

エンジニア視点で「実装しやすく、破綻しにくい」デザインのルールも固める。

4-1. グリッド & レイアウト

ページは基本：

上：固定ヘッダー height: 56px〜64px

左：サイドバー width: 220〜240px

右：コンテンツ領域（スクロール）

コンテンツの横幅：

最大 1200〜1280px に制限（横に広がりすぎると読みにくい）

余白：

ページ左右パディング 24px

コンポーネント内のパディング 16px ベース

4-2. タイポグラフィ

ざっくりルール：

ページタイトル（案件名）：font-size: 20〜24px、font-weight: 600

セクションタイトル（「タスク」「Issue」など）：16〜18px

通常テキスト：14〜16px

補足テキスト（サブ情報）：12〜13px、グレー

一貫して「案件名が一番強い」「タブ見出しが次に強い」。

4-3. コンポーネント定義（最小セット）

1. Button

primary：主要アクション（新規案件作成、保存など）

secondary：サブアクション（キャンセル、フィルタなど）

danger：削除、キャンセル

状態：default / hover / disabled / loading

2. Badge（ステータスラベル）

用途：Project.status / Task.status / Issue.status

色はステートで統一：

青系：進行中（in_progress / review）

緑系：完了（completed / done / closed）

橙系：警戒（on_hold / blocked / waiting）

赤系：危険・中止（canceled / high priority Issue）

グレー系：ドラフト・未着手（draft / todo / open）

3. ProgressBar（進捗バー）

Project ヘッダーと一覧で再利用

入力：value(0〜100)＋color（通常は一色でOK）

％数値ラベルを右側に常に表示

4. Table / List

案件一覧

Issue一覧

WorkLog集計

列ごとの揃え方：

テキスト：左寄せ

日付・数値：右寄せ

ステータス・担当者：中央 or 左寄せ（揃える）

5. Tabs

Project詳細の「概要 / タスク / Issue / 作業ログ / 引き継ぎ」

クリックで URL のクエリ or パスを切るようにして SPA としてわかりやすく。

6. SidePanel / Drawer

タスク詳細・Issue 詳細は、別ページ遷移ではなく右ペイン（SidePanel）で表示

この方が「一覧を見ながら詳細を触れる」ので、作業効率が高いし、実装もわかりやすい。

4-4. 状態デザイン（地味だけど超重要）

Empty State（データがないとき）

例：タスクが0件のとき

単なる「データがありません」ではなく、「＋ 新規タスクを作成」ボタンを中心に置く

Loading State

テーブルとパネルに Skeleton を出す

「いま読んでます」を明示する

Error State

通信失敗時：「再読み込みボタン」＋「問い合わせ先」を表示できるように

No-permission State（権限なし）

外注が金額や顧客詳細を見ようとしたとき

「権限がありません」とだけ出すと冷たいので、「担当PMに問い合わせてください」ぐらいは書いておく
